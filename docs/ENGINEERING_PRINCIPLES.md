# 엔지니어링 원칙 (Engineering Principles)

> "우리는 코드를 작성하는 것이 아니라, 가치를 창출합니다."

이 문서는 본 프로젝트의 모든 기술적 의사결정과 개발 문화의 근간이 되는 원칙을 정의합니다. 모든 팀원(및 AI 에이전트)은 이 원칙을 준수해야 합니다.

## 1. Product First (기획자 우선 사고)
**"Why"가 없는 기능은 만들지 않습니다.**
- **사용자 중심:** 기술적인 화려함보다 사용자가 얻을 실질적인 이득을 최우선으로 합니다.
- **역제안의 의무:** 요구사항이 비즈니스 목표와 상충되거나 더 나은 대안이 있다면, 단순히 수용하지 않고 더 나은 방향을 제시해야 합니다.
- **임팩트 평가:** 모든 개발 작업 전에 "이것이 비즈니스에 어떤 임팩트를 주는가?"를 스스로 묻고 답할 수 있어야 합니다.

## 2. Pragmatic Tech Choice (실용적 기술 선택)
**기술은 도구일 뿐, 목적이 아닙니다.**
- **Technology Agnostic:** 특정 언어나 프레임워크에 대한 종교적 신념을 배제합니다. 문제 해결에 가장 적합한 도구를 선택합니다.
- **Boring Technology:** 검증되지 않은 최신 기술(Hype)보다는, 실패 가능성이 낮고 생태계가 성숙한 '지루한 기술'을 선호합니다.
- **Over-engineering 금지:** YAGNI (You Aren't Gonna Need It) 원칙을 철저히 지킵니다. 현재 필요한 문제 해결에 집중하되, 미래의 확장성을 위한 최소한의 유연성만 열어둡니다.

## 3. Veteran Craftsmanship (장인의 개발 역량)
**프로는 디테일에서 차이를 만듭니다.**
- **Production-Ready:** "내 로컬에서는 되는데요"는 허용되지 않습니다. 배포 가능한 수준의 견고함(Error Handling, Security, Logging)을 갖춰야 합니다.
- **Readable Code:** 코드는 기계가 읽기 전에 사람이 읽는 글입니다. 주석과 명확한 변수명으로 의도를 충분히 설명해야 합니다.
- **Responsibility:** 자신이 작성한 코드에 대해 끝까지 책임을 집니다. 테스트 코드는 선택이 아닌 필수입니다.

## 4. Communication Protocol (소통 원칙)
- **한국어 우선:** 모든 문서, 커밋 메시지, 이슈 토론은 **한국어**로 진행하여 명확한 소통을 지향합니다. (단, 코드 내 변수명 등은 영문 표준을 따름)
- **3단계 사고 프로세스:**
    1. **Planner View:** 무엇을, 왜 만드는가? (기획)
    2. **Architect View:** 어떻게 구조를 잡을 것인가? (설계)
    3. **Developer View:** 어떻게 구현할 것인가? (구현)

---
*이 원칙은 불변의 진리가 아니며, 프로젝트의 성장과 함께 진화할 수 있습니다.*
